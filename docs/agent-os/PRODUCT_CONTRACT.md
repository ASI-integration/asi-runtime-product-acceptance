# ASI Agent OS v0 — product contract

## Назначение

Этот документ фиксирует инварианты продукта, которые агент обязан сохранять при автономной работе. Он не заменяет задачу, архитектурные документы или acceptance-критерии. Если источники расходятся, агент останавливается и регистрирует blocker.

## Продуктовая цель

ASI помогает объектам размещения снижать зависимость от OTA и автоматизировать операции объекта. Приоритет — рынок РФ, простые русскоязычные сценарии и постепенное развитие поверх уже работающих контуров.

## Неприкосновенные инварианты

- Не менять утверждённый UX без явного разрешения владельца.
- Не менять публичные продуктовые обещания, цены, платежную логику или юридически значимые формулировки без явного разрешения владельца.
- Не раскрывать секреты, персональные данные, внутренние диагностические данные или служебные идентификаторы в публичном интерфейсе и отчётах.
- Не включать реальные исходящие сообщения или вызовы внешних провайдеров как побочный эффект разработки либо acceptance.
- Сохранять изоляцию аккаунтов и объектов; не ослаблять авторизацию, RLS, account scope и server-only границы.
- Сохранять существующие доменные seams и SSOT. Новый параллельный lifecycle, intake, task или communication flow требует отдельного архитектурного решения владельца.
- Считать migration-файлы append-only историей. Изменение уже применённой migration требует отдельного решения и плана совместимости.

## Защищённая область location

Корневой `AGENTS.md` остаётся основным ограничителем для location-модуля:

- scoring, SSOT, `LocationDecision` и `LocationPublicSummary` чувствительны;
- H3 diagnostics не влияют на `finalScore` без явно поставленной задачи;
- raw diagnostics не попадают в публичный RU UI;
- следующий шаг должен быть инкрементальным: diagnostics, затем soft gates, затем visuals платного отчёта.

## Операционный контракт

- `main` — source branch релиза, но merge сам по себе не означает production deploy.
- Production работает через ручной artifact workflow, описанный в `DEPLOY.md` и `.github/workflows/deploy.yml`.
- Staging обязан оставаться отделённым по пути, процессу, порту, конфигурации, секретам и базе данных.
- Deploy не применяет migrations автоматически.
- Release идентифицируется полным commit SHA; health/version checks должны подтверждать ожидаемый SHA.
- Production deploy, production migrations, production data mutations, secrets, DNS, платежи и изменения утверждённого UX всегда относятся к red-действиям.

## Правило изменения контракта

Изменение этого файла является yellow только для уточнения уже подтверждённого поведения. Любое изменение продуктового инварианта или перевод red-действия в более низкий класс требует явного разрешения Николая и отдельного review.
